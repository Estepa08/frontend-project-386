# Video Meeting Booking API

## Описание приложения
SPA для бронирования видеовстреч. Две роли: **Admin** (организатор) и **User** (клиент).

### Сущности (из `main.tsp`)
- **Admin** — id, name, email, рабочие часы, типы встреч
- **User** — id, name, email
- **MeetingType** — длительность (15/30 мин), категория (single/group/private), видимость, гостевые инвайты
- **Meet** — связка Admin + User + MeetingType, статус (confirmed/cancelled), inviteLink, тема, гости

### Сценарий использования
1. Admin регистрируется и настраивает график работы + типы встреч
2. User регистрируется, смотрит доступные даты/слоты админа
3. User бронирует встречу
4. Обе стороны могут просматривать и управлять встречами

## Стандарты кода

Следуй этим правилам при написании и ревью кода в проекте. Каждое правило — устоявшийся индустриальный стандарт, не произвольное предпочтение. При ревью явно указывай, какому пункту соответствует или не соответствует решение, и предлагай альтернативу с обоснованием (простота / распространённость / масштабируемость / читаемость).

### 1. Separation of Concerns

Разделяй код на слои: сеть (API-вызовы) → логика (хуки, стейт-менеджмент) → UI (компоненты). Данные идут только в одну сторону — сверху вниз по слоям, импорты снизу вверх запрещены.

Единица ответственности — модуль, а не файл. Один модуль может содержать несколько тесно связанных мелких функций/типов, если они выражают одну идею.

Источники: Clean Architecture (R. Martin), Feature-Sliced Design, TanStack Query (queryFn → useQuery → Component).

### 2. Validation & Safety

- Всегда проверяй `response.ok` / статус ответа перед использованием данных.
- Валидируй входные данные через схему (Zod), не через разрозненные `if` в компонентах.
- Проверки доступа делай по роли (`user.role`), а не по `user.id`. Guard по ID — классическая уязвимость broken access control (OWASP A01).
- Никаких захардкоженных значений — только конфиг/переменные окружения (12-Factor App).
- Для мутаций явно обрабатывай `onError`, для запросов — `isError`.

### 3. Naming

- Множественное число для массивов/коллекций, единственное — для объектов.
- `set` = полная замена значения, `update` = частичное изменение. Не смешивай.
- Не используй сокращения — имена должны быть произносимыми и описательными (Google TS Style Guide, Clean Code).
- В деструктуризации/параметрах пиши `state`, а не `s`.
- Однобуквенные переменные (`x`, `i`, `_`) допустимы в коротких (однострочных) коллбэках, например в `.map()`. Если тело лямбды длиннее одной строки — используй осмысленное имя, отражающее суть элемента.

### 4. Readability

- Ориентир — не более ~150–200 строк на файл (ESLint `max-lines`, гибко в зависимости от задачи).
- Соблюдай DRY.
- Не используй эмодзи в коде — командная конвенция, но в продакшене соблюдай её строго.
- См. правило про однобуквенные переменные (п.3).

### 5. Data Flow

- По умолчанию — однонаправленный поток данных (Redux/Flux/React-модель, в этом проекте — Zustand + TanStack Query). Это доминирующий, но не единственный паттерн — явно указывай, если решение отходит от этой модели и почему.
- Соблюдай dependency rule: зависимости указывают внутрь (UI зависит от логики, логика — от данных, не наоборот).
- Хуки и сервисы не должны знать о UI-слое (см. API TanStack Query / RTK Query как эталон).

### 6. Process

- Schema-first / contract-first: сначала описывается контракт (`main.tsp` → OpenAPI → `openapi-typescript`), потом реализация.
- Спецификация и критерии приёмки — до написания кода (TDD/BDD/acceptance criteria), а не после.

### 7. Conventional Commits

Формат коммитов для Release Please:

```
<type>(<scope>): <описание>
```

**Типы и влияние на версию (SemVer):**

| Type | Bump | Changelog |
|------|------|-----------|
| `feat` | minor | Features |
| `fix` | patch | Bug Fixes |
| `perf` | patch | Performance Improvements |
| `refactor` | patch | Refactors |
| `test` | patch | Tests |
| `ci` | patch (no release) | Continuous Integration |
| `docs` | — (hidden) | — |
| `chore` | — (hidden) | — |

**Scope** указывает пакет:
- `feat(client):` — только `client/`
- `fix(server):` — только `server/`
- `feat:`, `fix:` (без scope) — изменения в корне (opencode.json, Makefile, .github/), релиз обоих пакетов.

Breaking change — суффикс `!` или `BREAKING CHANGE:` в теле коммита.

**Правила для агента:**
- Каждый коммит начинается с типа и scope
- Scope обязателен, если изменения только в одном пакете
- BREAKING CHANGE только при сознательном ломающем изменении публичного API
- Не смешивать изменения разных пакетов в одном коммите
